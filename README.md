# Event WebAPI

Простая реализация REST API на ASP.NET core для управления локальными списками событий.

## Используемые технологии:
* .Net 10
* ASP.NET core Web Api (Microsot)
* Swashbuckle (Swagger)
* xunit
* Moq


## Запуск проекта
1. Склонировать репозиторий
2. Запустить приложение dotnet run
3. Для Swagger использовать путь: localhost:\[port]/swagger


## Основные методы для работы с Event
| Метод     | Эндпоинт              | Описание                          | Ответ                            |
| --------- | --------------------- |  -------------------------------- | -------------------------------- |
| GET       | `/api/events`         | Получение списока всех событий    | 200 OK: Возвращает объект PaginatedResult<EventResponse>          |
| GET       | `/api/events/{id}`    | Получение события по ID           | 200 OK: Возвращает объект EventResponse для найденного события    |
| POST      | `/api/events`         | Создать новое событие             | 201 Created: Событие успешно создано                              |
| PUT       | `/api/events/{id}`    | Обновление существующего события  | 200 OK: Возвращает обновленный объект EventResponse               |
| DELETE    | `/api/events/{id}`    | Удаление события по ID            | 204 No Content: Событие успешно удалено. Тело ответа отсутствует  |

Получение списка событий (с фильтрацией и пагинацией)
`GET /api/events`
Возвращает список событий, обернутый в объект пагинации. Все параметры являются опциональными.

Параметры запроса:
* title (string): Фильтрация по названию (частичное совпадение, регистронезависимо).
* from (DateTime): Выбирает события, начинающиеся не ранее этой даты.
* to (DateTime): Выбирает события, заканчивающиеся не позднее этой даты.
* page (int): Номер страницы (по умолчанию 1).
* pageSize (int): Количество элементов на страницу (по умолчанию 10).

## Обработка ошибок
В проекте реализована глобальная обработка исключений через Middleware. В случае возникновения ошибки сервер возвращает единообразный ответ
Типовые коды ответов:
* 400 Bad Request: Ошибки валидации (например, дата окончания раньше даты начала).
* 404 Not Found: Ресурс с указанным ID не найден в системе.
* 500 Internal Server Error: Непредвиденная ошибка на стороне сервера.

## Основные методы для работы с Booking
| Метод     | Эндпоинт                       | Описание                                    | Ответ                                                                                            |
| --------- | ------------------------------ |  ------------------------------------------ | -------------------------------------------------------------------------------------------------|
| GET       | `/bookings/{id:guid}`          | Получение конкретного бронирования          | 202 Accepted. В теле возвращается объект брони, в заголовке Location, ссылка на статус брони     |
| POST      | `/events/{eventId:guid}/book`  | Создание нового бронирования по Id события  | 200 OK. Объект бронирования                                                                      |

Модель Booking:
- ID (Guid): Уникальный идентификатор
- EventId (Guid): ID события, к которому относится бронь
- Status (Enum): Статус бронирования 
- CreatedAt (DateTime): Время создания заявки
- ProcessedAt (DateTime?): Время обработки фоновым сервисом

Статусы бронирования:
- Pending (0): Заявка создана и ожидает обработки 
- Confirmed (1): Бронирование успешно подтверждено 
- Rejected (2): Бронирование отклонено
- 
## Фоновая обработка
В системе реализован BackgroundBookingService на базе BackgroundService, который взаимодействует с данынми
- Раз в несколько секунд сервис опрашивает IBookingRepository на наличие записей со статусом Pending
- Для каждой найденной записи выполняется изменение статуса меняется на Confirmed 
- После обработки обновляется поле ProcessedAt
- Изменения сохраняются

## Тестирование
Для обеспечения надежности бизнес-логики написаны Unit-тесты с использованием xUnit.

Запуск тестов:
`dotnet test <путь до проекта>`
Например: `dotnet test C:\Projects\MyUnitTests\MyUnitTests.csproj`

Запуск тестов по имени:
`dotnet test C:\Projects\MyUnitTests\MyUnitTests.csproj --filter EventsServiceTests`

Для запуска всех тестов выполните команду в корневой папке решения или в папке тестового проекта:
Тесты покрывают следующие сценарии:
* CRUD операции (создание, чтение, обновление, удаление).
* Сложная фильтрация (по тексту и датам).
* Логика пагинации (корректный расчет Skip/Take).
* Негативные сценарии (неверные ID, некорректные даты).
